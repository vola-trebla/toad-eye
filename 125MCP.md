## Executive Summary

MCP (Model Context Protocol) — это будущее инструментов для LLM-агентов. toad-eye уже трейсит _клиентскую_ сторону (OpenAI/Anthropic/Gemini SDK), но не видит, что происходит _внутри_ MCP серверов, когда LLM вызывает tools. Задача — создать `toadEyeMiddleware()`, который MCP-сервер автор вставляет одной строкой и получает автоматический tracing + metrics для каждого `tools/call`, `resources/read`, `prompts/get`.

**Ключевое отличие от существующих инструментаций:** MCP — это stateful JSON-RPC 2.0 протокол, а не REST. Нельзя просто monkey-patch'ить `fetch`. Нужен **Wrapper Pattern** — оборачиваем `McpServer`/`Server` instance и перехватываем хэндлеры.

---

## 1. Архитектурное решение: Wrapper Pattern

### Почему НЕ monkey-patching

В отличие от OpenAI SDK, где мы патчим `OpenAI.Chat.Completions.prototype.create`, MCP SDK использует:

- Private class members (`_requestHandlers` — приватный Map)
- Strict TypeScript typing
- Нет стабильного публичного прототипа для перехвата

Research подтверждает: **Wrapper Pattern** — единственный стабильный путь. Есть proposal SEP-1763 (Interceptor Framework), но он ещё не ратифицирован.

### Как это работает

```
User Code                    toad-eye                     MCP SDK
─────────                    ─────────                    ─────────
server.tool("calc", ...)  →  intercept .tool() call   →  original .tool() registered
                             wrap handler with span       with wrapped handler

Client calls tools/call   →  SDK routes to handler    →  wrapped handler executes:
                                                         1. Start OTel span
                                                         2. Extract _meta.traceparent
                                                         3. Run original handler
                                                         4. Record duration + result
                                                         5. End span
```

## 2. OTel Semantic Mapping

Маппинг MCP операций → OTel GenAI semconv (v1.40.0+):

| MCP Method               | `gen_ai.operation.name` | Span Name                  | Key Attributes                                         |
| ------------------------ | ----------------------- | -------------------------- | ------------------------------------------------------ |
| `tools/call`             | `execute_tool`          | `execute_tool {tool_name}` | `gen_ai.tool.name`, `gen_ai.tool.call.id`              |
| `resources/read`         | `retrieval`             | `retrieval {uri}`          | `gen_ai.data_source.id`, `gen_ai.retrieval.query.text` |
| `prompts/get`            | `prompt`                | `prompt {name}`            | `gen_ai.prompt.name`                                   |

> **Важно:** span names уже align с нашим существующим `agent.ts` (`execute_tool`, `chat`). Трейсы из MCP сервера будут seamlessly соединяться с agent spans.

## 4. User-Facing API

### Минимальный use case (одна строка)

```typescript
import { McpServer } from "@modelcontextprotocol/sdk/server/index.js";
import { toadEyeMiddleware } from "toad-eye/mcp";

const server = new McpServer({ name: "my-server", version: "1.0.0" });

// ✅ Drop-in — одна строка
toadEyeMiddleware(server);

// Регистрируем tools как обычно — они уже инструментированы
server.tool("calculate", { expression: z.string() }, async ({ expression }) => {
  return { content: [{ type: "text", text: String(eval(expression)) }] };
});
```

### Полная конфигурация

```typescript
import { toadEyeMiddleware } from "toad-eye/mcp";
import type { ToadMcpOptions } from "toad-eye/mcp";

const options: ToadMcpOptions = {
  // OTel config (если initObservability() уже вызван — не нужно)
  serviceName: "my-mcp-server", // default: берёт из server.name
  endpoint: "http://localhost:4318", // OTLP endpoint

  // Privacy
  recordInputs: false, // default: false — НЕ записываем tool arguments
  recordOutputs: false, // default: false — НЕ записываем tool results
  redactKeys: ["apiKey", "token"], // keys to redact from arguments
  maxPayloadSize: 4096, // truncate large payloads (bytes)

  // Metrics
  captureMetrics: true, // default: true

  // Advanced
  propagateContext: true, // extract traceparent from _meta
  onToolCall: (span, toolName) => {}, // hook for custom attributes
};

toadEyeMiddleware(server, options);
```

## 5. TypeScript Interfaces

```typescript
// packages/instrumentation/src/mcp/types.ts

import type { Span } from "@opentelemetry/api";

export interface ToadMcpOptions {
  /** Service name for OTel resource. Defaults to MCP server name. */
  serviceName?: string;

  /** OTLP endpoint. Uses existing OTel config if initObservability() was called. */
  endpoint?: string;

  /** Record tool call arguments in spans. Default: false (privacy). */
  recordInputs?: boolean;

  /** Record tool call results in spans. Default: false (privacy). */
  recordOutputs?: boolean;

  /** Keys to redact from recorded arguments. */
  redactKeys?: string[];

  /** Max payload size in bytes before truncation. Default: 4096. */
  maxPayloadSize?: number;

  /** Emit OTel metrics. Default: true. */
  captureMetrics?: boolean;

  /** Extract W3C traceparent from _meta field. Default: true. */
  propagateContext?: boolean;

  /** Hook called on each tool call span. Add custom attributes here. */
  onToolCall?: (span: Span, toolName: string, args: unknown) => void;

  /** Hook called on each resource read span. */
  onResourceRead?: (span: Span, uri: string) => void;
}

export interface McpSpanAttributes {
  "gen_ai.operation.name": string;
  "gen_ai.tool.name"?: string;
  "gen_ai.tool.call.id"?: string;
  "gen_ai.data_source.id"?: string;
  "gen_ai.prompt.name"?: string;
  "gen_ai.request.model"?: string;
  "mcp.server.name": string;
  "mcp.server.version": string;
  "mcp.transport": "stdio" | "sse" | "streamable-http";
  "mcp.session.id"?: string;
}
```

## 7. Context Propagation

### HTTP/SSE Transport

Стандартные `traceparent` / `tracestate` headers — работают как обычно.

### STDIO Transport (⚠️ сложность)

В stdio нет HTTP headers. Context передаётся через `_meta` поле в JSON-RPC params:

```json
{
  "jsonrpc": "2.0",
  "method": "tools/call",
  "params": {
    "name": "calculate",
    "arguments": { "expression": "2+2" },
    "_meta": {
      "traceparent": "00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01"
    }
  }
}
```

```typescript
// src/mcp/context.ts
import { context, propagation } from "@opentelemetry/api";

export function extractContextFromMeta(meta?: Record<string, unknown>) {
  if (!meta?.traceparent) return context.active();

  const carrier = {
    traceparent: String(meta.traceparent),
    ...(meta.tracestate ? { tracestate: String(meta.tracestate) } : {}),
  };

  return propagation.extract(context.active(), carrier);
}
```

> **Risk:** Не все MCP hosts поддерживают `_meta` injection. Если `traceparent` нет — создаём новый root span. Трейс не ломается, просто не linked.

## 8. STDIO Isolation (⚠️ Critical Risk)

**Проблема:** В stdio mode, `stdout` — это JSON-RPC transport. Если toad-eye (или OTel SDK) случайно пишет в stdout через `console.log` или misconfigured logger — **MCP connection крашится**.

**Решение:**

```typescript
// В middleware.ts при инициализации
import { diag, DiagConsoleLogger, DiagLogLevel } from "@opentelemetry/api";

function initSafeLogging() {
  // Redirect OTel diagnostics to stderr only
  diag.setLogger(new DiagConsoleLogger(), DiagLogLevel.WARN);
  // Override console.log in the OTel context to use stderr
}
```

Плюс в документации: **"Never use console.log in stdio MCP servers. Use console.error for debugging."**

| MCP SDK breaking changes (pre-1.0 vibes)   | 🟡 Medium | Pin peer dep `>=1.0.0`, integration tests    |
